# 디자인 패턴 (Design Pattern)

## 1. 디자인 패턴이란?

디자인 패턴이란 **객체지향 소프트웨어 설계 과정에서 반복적으로 등장하는 문제에 대해 검증된 해결 방법을 정리한 설계 템플릿**입니다.  
단순한 코드 조각이 아니라, **클래스와 객체의 역할·책임·협력 관계를 구조화하는 설계 방법론**에 가깝습니다.

> 디자인 패턴의 목적은 “코드를 잘 짜는 것”이 아니라  
> **변경에 강하고, 유지보수가 쉬우며, 확장 가능한 구조를 만드는 것**입니다.

---

## 2. 디자인 패턴의 특징

- **재사용 가능**: 여러 프로젝트에서 반복 적용 가능
- **의사소통 도구**: 설계 의도를 패턴 이름으로 명확히 전달
- **언어 비의존적**: 특정 언어 문법이 아닌 구조와 역할 중심
- **객체지향 원칙 기반**: SOLID 원칙과 밀접한 관계

---

## 3. GoF 디자인 패턴

일반적으로 디자인 패턴이라 하면 **GoF(Gang of Four) 디자인 패턴**을 의미합니다.  
Erich Gamma 외 4명이 정리한 **23개의 객체지향 디자인 패턴**입니다.

### 분류 기준
- **생성 (Creational)**: 객체 생성 방식
- **구조 (Structural)**: 객체/클래스의 조합 구조
- **행위 (Behavioral)**: 객체 간 책임과 행위 분배

---

## 4. 생성 패턴 (Creational Patterns)

객체 생성 로직을 캡슐화하여 **결합도를 낮추고 생성 과정을 유연하게 관리**합니다.

| 패턴 | 핵심 개념 |
|---|---|
| Singleton | 인스턴스를 하나만 유지 |
| Factory Method | 객체 생성을 서브클래스에 위임 |
| Abstract Factory | 관련 객체군을 일관되게 생성 |
| Builder | 복잡한 객체를 단계적으로 생성 |
| Prototype | 기존 객체를 복제하여 생성 |

---

## 5. 구조 패턴 (Structural Patterns)

객체와 클래스의 관계를 구조화하여 **확장성과 유연성**을 높입니다.

| 패턴 | 핵심 개념 |
|---|---|
| Adapter | 인터페이스 변환 |
| Bridge | 추상과 구현 분리 |
| Composite | 트리 구조 표현 |
| Decorator | 기능을 동적으로 확장 |
| Facade | 복잡한 서브시스템 단순화 |
| Flyweight | 객체 공유로 메모리 절약 |
| Proxy | 대리 객체를 통한 접근 제어 |

---

## 6. 행위 패턴 (Behavioral Patterns)

객체 간의 책임과 커뮤니케이션 방식을 정의합니다.

| 패턴 | 핵심 개념 |
|---|---|
| Strategy | 알고리즘을 교체 가능하게 캡슐화 |
| Observer | 상태 변경 알림 |
| Command | 요청을 객체로 캡슐화 |
| State | 상태에 따른 행위 변경 |
| Template Method | 알고리즘 골격 정의 |
| Iterator | 컬렉션 순회 방법 분리 |
| Mediator | 객체 간 직접 의존 제거 |
| Memento | 상태 저장/복원 |
| Chain of Responsibility | 책임을 체인으로 전달 |
| Visitor | 기능과 구조 분리 |
| Interpreter | 문법 해석 |

---

## 7. 디자인 패턴과 SOLID의 관계

디자인 패턴은 SOLID 원칙을 **구체적인 설계 형태로 구현한 결과물**입니다.

- **SRP** → Strategy, Command
- **OCP** → Factory, Decorator
- **LSP** → Template Method
- **ISP** → Adapter
- **DIP** → Abstract Factory, Bridge

---

## 8. 디자인 패턴을 사용할 때 주의점

- 패턴은 **목적이 아닌 수단**
- 문제 없이 단순한 구조에 패턴 적용은 과설계
- 패턴 이름보다 **문제 상황과 의도 이해가 중요**
- “이 패턴을 써야 한다”가 아니라  
  **“이 구조가 왜 필요한가”를 먼저 고민**

---

## 9. 정리

- 디자인 패턴은 **경험이 축적된 설계 지식의 집합**
- 코드 수준이 아닌 **구조와 역할 관점**
- 유지보수·확장·협업을 고려한 설계 도구
- 패턴을 외우는 것보다 **언제, 왜 쓰는지 이해하는 것이 핵심**
